类模板的模板参数推导

C++17填补了模板推导规则中的一个空白。现在，模板参数推导不仅适用于函数模板，也适用于类模板。这也意味着，许多使用make_Type函数的代码现在可以省略了。

C++17 的 CTAD（Class Template Argument Deduction，类模板实参推导）就是: 编译器会根据构造函数，自动推导出模板参数。

一、CTAD 是解决什么问题的？

在 C++17 之前：

1

std::pair<int, int> p(1, 2);

C++17 之后可以写：

1

std::pair p(1, 2);  // 自动推导

👉 编译器会根据构造函数，自动推导出模板参数。

二、CTAD 的核心机制

本质上：

👉 编译器会“假装”把构造函数变成一个函数模板来做推导

举个例子：

1
2
3
4

template<typename T>
struct A {
    A(T t);
};

你写：

1

A a(10);

编译器会“脑补”成：

1

A(T) -> A<T>

然后像函数模板一样推导：

1

T = int

👉 最终得到：

1

A<int> a(10);

三、推导来源（非常关键）

CTAD 的推导来源有两种：

1️⃣ 隐式推导（来自构造函数） 
1
2
3
4
5
6

template<typename T>
struct Box {
    Box(T v);
};

Box b(10);  // 推导出 Box<int>

👉 这是最常见情况

2️⃣ 显式推导（你自己写的 guide）

有些情况，编译器推不出来 or 推得不对，这时就要你手动写推导指引（Deduction Guides）

案例: 字符串坑

1
2
3
4
5
6
7

template<typename T>
struct Box {
    T value;
    Box(T v) : value(v) {}
};

Box b("hello");

👉 默认推导：

1

Box<const char*>  // ❌ 很多时候你不想要这个

👉 加推导指引：

1

Box(const char*) -> Box<std::string>;

👉 现在：

1
2

Box b("hello");  
// -> Box<std::string> ✅
